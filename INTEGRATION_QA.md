# Whisper API для интегратора

## 1. Формат передачи данных

Текущий API ИИ - сервера поддерживает два варианта:

- `POST /transcribe` - загрузка готового аудиофайла по HTTP.
- `WebSocket /stream` - отправка бинарных аудиочанков по WebSocket.

gRPC сейчас не поддерживается.

Важно по WebSocket: это не настоящий real-time ASR с промежуточными
partial-результатами. Клиент отправляет бинарные чанки, затем отправляет текстовое
сообщение `END` или закрывает соединение. Сервер склеивает полученное аудио и
только после этого один раз запускает Whisper.

Рекомендация:

- Если есть уже готовые фрагменты аудио - использовать `POST /transcribe`.
- Если аудио идет во время звонка - нарезать его на фразы/сегменты по паузам и
  отправлять каждый сегмент через WebSocket или HTTP.
- Не держать один бесконечный WebSocket на весь многочасовой поток без `END`:
  текущий сервер буферизует аудио в памяти до завершения сегмента.

Внешние URL, если доступ идет через nginx:

```text
GET  http://172.16.0.16/whisper/healthz
POST http://172.16.0.16/whisper/transcribe
WS   ws://172.16.0.16/whisper/stream?api_key=<WHISPER_API_KEY>
```

Прямой доступ к контейнеру/host port:

```text
GET  http://172.16.0.16:8001/healthz
POST http://172.16.0.16:8001/transcribe
WS   ws://172.16.0.16:8001/stream?api_key=<WHISPER_API_KEY>
```

## 2. Формат аудио и ресемплинг

Да, аудио нужно приводить к `16 kHz` перед отправкой.

Для WebSocket:

```text
PCM float32
mono
16000 Hz
raw bytes
```

Для HTTP `/transcribe`:

```text
multipart/form-data
field: file
audio file readable by soundfile/libsndfile
sample rate: 16000 Hz
```

Если отправить HTTP-файл с другим sample rate, сервер вернет `400`:

```text
Ожидается 16kHz, пришло <sr>
```

Если исходный звук телефонии идет в `8 kHz`, `44.1 kHz`, `48 kHz` или другом
sample rate, ресемплинг нужно делать на стороне клиента/интегратора.

## 3. Каналы: оператор и абонент

Текущий Whisper API ожидает один mono-поток. Он не делает diarization и не
размечает, кто говорит.

Варианты:

- Один mono-поток со смешанным оператором и абонентом - можно использовать, если
  нужен только общий текст без разделения ролей.
- Один stereo-поток, где левый/правый канал обозначают стороны, сейчас делать не стоит. 
Текущий API не сохраняет и не интерпретирует каналы как роли.
- Два отдельных mono-потока - лучший вариант, если важно сохранить роли
  `operator` / `abonent`.

Рекомендации для интеграции с ролями:

1. Захватывать оператора и абонента раздельно.
2. Каждый канал приводить к `mono float32 16 kHz`.
3. Отправлять каждый канал отдельными сегментами.
4. На стороне интегратора хранить `call_id`, `channel`, `operator_id`,
   timestamp начала сегмента и timestamp конца сегмента.
5. После ответа Whisper склеивать итоговый диалог по timestamps.

Сейчас сам Whisper API не принимает и не возвращает `call_id`, `operator_id` или
`channel`. Если нужно, чтобы эти поля проходили через API и возвращались в
ответе, потребуется небольшая доработка endpoint с нашей стороны.

## 4. Тишина и паузы

На сервере включен VAD внутри `faster-whisper`, но он применяется уже во время
распознавания полученного сегмента. Он не решает задачу бесконечного входящего
потока и не освобождает клиента от сегментации.

Рекомендации:

- Не отправлять постоянную тишину бесконечным потоком.
- На стороне интегратора делать простую voice activity segmentation.
- Завершать сегмент после паузы, например после `0.5-1.0` секунды тишины.
- После завершения сегмента отправлять `END` для WebSocket или отдельный HTTP
  request для `/transcribe`.
- Если динамики дают звук только во время звонка, то вне звонка ничего
  отправлять не нужно.
- Если один из каналов молчит, можно не отправлять пустой сегмент.

Текущий серверный VAD использует `VAD_MIN_SILENCE=0.5` секунды по умолчанию.

## 5. Идентификация оператора и звонка

Для самого распознавания `operator_id` и `call_id` не нужны: API stateless (не хранит состояния) и
распознает только переданное аудио.

Но для интеграции с CRM/телефонией идентификаторы нужны. Рекомендуемый минимум на
стороне интегратора:

```json
{
  "call_id": "unique-call-id",
  "operator_id": "operator-id",
  "channel": "operator|abonent",
  "segment_id": 1,
  "started_at": "2026-06-11T12:00:00Z",
  "audio_start_ms": 1200,
  "audio_end_ms": 4800
}
```

Текущий API эти поля не сохраняет и не возвращает. Их нужно хранить рядом с
запросом на стороне интегратора. Если нужно, чтобы API принимал metadata, можно
добавить query-параметры или JSON envelope в отдельном endpoint.

## Авторизация

HTTP `/transcribe`:

```http
X-API-Key: <WHISPER_API_KEY>
```

WebSocket `/stream`:

```text
?api_key=<WHISPER_API_KEY>
```

## Пример HTTP-запроса

```bash
curl -s "http://172.16.0.16/whisper/transcribe?task=transcribe&language=ru&beam_size=5&vad=true" \
  -H "X-API-Key: <WHISPER_API_KEY>" \
  -F "file=@segment_16khz_mono.wav"
```

## Пример WebSocket-протокола

```text
CONNECT ws://172.16.0.16/whisper/stream?api_key=<WHISPER_API_KEY>
SEND binary chunk 1: float32 mono 16 kHz
SEND binary chunk 2: float32 mono 16 kHz
SEND text: END
RECEIVE json response
CLOSE
```

Ответ:

```json
{
  "model": "large-v3",
  "device": "cuda",
  "compute_type": "float16",
  "language": "ru",
  "language_probability": 1.0,
  "time_sec": 0.123,
  "text": "распознанный текст",
  "segments": [
    {
      "start": 0.0,
      "end": 2.4,
      "text": "распознанный текст"
    }
  ]
}
```

## Все те же рекомендации кратко

Для первой версии интеграции:

1. Делать клиентскую сегментацию по паузам.
2. Ресемплить каждый сегмент в `mono 16 kHz`.
3. Если роли не нужны - отправлять один смешанный mono-поток.
4. Если роли нужны - отправлять оператора и абонента отдельными mono-сегментами.
5. Хранить `call_id`, `operator_id`, `channel` и timestamps на стороне
   интегратора.
6. Использовать HTTP `/transcribe` для готовых сегментов или WebSocket `/stream`
   для передачи чанков одного сегмента.
